## 1. Thông tin tổng quan
- **Category:** Reverse Engineering
- **Difficulty:** Medium / Hard

## 2. Đề bài
Bài toán cung cấp một file mã nguồn C cực kỳ đơn giản `checker.c` và một bộ trình biên dịch `tcc` (Tiny C Compiler) được build sẵn, cùng với thư mục `runtime`.

`checker.c`:
```c
#include <stdio.h>
#include <string.h>

/* Our build system links this from the internal audit library. */
extern int audit(const char *answer);

int main(void)
{
    char answer[128];
    puts("Build provenance check");
    fputs("receipt: ", stdout);
    if (!fgets(answer, sizeof answer, stdin))
        return 1;
    answer[strcspn(answer, "\n")] = 0;
    puts(audit(answer) ? "accepted" : "rejected");
    return 0;
}
```
Lệnh biên dịch theo mặc định: `./tcc -B./runtime checker.c -o checker`

## 3. Quá trình phân tích

Đọc mã nguồn `checker.c`, ta thấy chương trình gọi hàm `audit(answer)` vốn chỉ được khai báo `extern` chứ không có file nào chứa định nghĩa cụ thể. Tuy nhiên, khi biên dịch bằng trình biên dịch `tcc` đính kèm, file thực thi `checker` sinh ra lại chạy mượt mà không hề bị lỗi "undefined reference". 

Đây là mô phỏng lỗ hổng kinh điển **Trusting Trust (Ken Thompson's Hack)**. Trình biên dịch `tcc` đã bị cấy backdoor. Khi nhận diện đang biên dịch file `checker.c`, nó tự động tiêm (inject) mã máy của hàm `audit()` thẳng vào file binary đầu ra.

**Phân tích hàm `audit` trong file thực thi `checker` sinh ra:**
1. **Keygen (Tự băm):** Hàm đọc chính file thực thi của nó qua `/proc/self/exe` và băm nội dung để lấy một giá trị làm khóa. Việc này diễn ra độc lập với input đầu vào của người dùng.
2. **Decrypt Payload:** Khóa thu được dùng để XOR-decrypt một blob dữ liệu 512-byte tại địa chỉ cố định `0x405b94` đưa vào một vùng nhớ R+X (Read + Execute). 
   - Cấu trúc blob sau khi giải mã: Header (bao gồm chiều dài chuỗi giải mã là $51$, marker `0x51` 'Q') và theo sau là $51$ instruction của một máy ảo (Custom VM), mỗi instruction dài 7 byte.
3. **Custom VM:** VM này chạy lặp qua từng ký tự của chuỗi input, liên tục tính toán Rolling Hash $h$ (khởi tạo mốc ở `0xC0DEC0DE`). Nó dùng 1 instruction 7-byte xử lý cho 1 ký tự:
   - Instruction format: `[0xa7 marker][b1][b2][target LE32]`
   - Thuật toán Hash:
     ```python
     h ^= (answer[i] + b1)
     h  = ROL32(h, b2 & 31)
     h += (i * 0x045d9f3b) ^ 0x9e3779b9
     ```
   - Cuối vòng lặp sẽ kiểm tra mốc `h == target[i]`.

**Hướng giải quyết:**
Thuật toán Rolling Hash của VM là chuỗi các phép tính XOR, Cộng/Trừ và Xoay bit (ROL/ROR) hoàn toàn có thể đảo ngược được. Bắt đầu từ giá trị `target` sau cùng, ta lùi ngược từng bước (`h_prev`), thực hiện undo phép cộng, dùng ROR (xoay phải) thay cho ROL (xoay trái) và gỡ XOR để tính ra chính xác chuỗi `answer` gốc.

## 4. PoC

Ta trích xuất file `blob.bin` đã giải mã sẵn ở runtime (bằng lệnh gdb dump binary memory) rồi dùng script để đảo ngược thuật toán VM như sau:

`solve.py`:
```python
#!/usr/bin/env python3

BLOB = open('blob.bin', 'rb').read()
M = 0xffffffff

# Hàm xoay phải (ROR) 32-bit
def ror(x, r):
    r &= 31
    return ((x >> r) | (x << (32 - r))) & M if r else x & M

def solve():
    n = BLOB[0] | (BLOB[1] << 8)
    assert BLOB[2] == 0x51
    h_prev = 0xC0DEC0DE
    ans = bytearray()
    
    for i in range(n):
        base = 3 + 7 * i
        assert BLOB[base] == 0xA7
        b1, b2 = BLOB[base + 1], BLOB[base + 2]
        target = int.from_bytes(BLOB[base + 3:base + 7], 'little')
        
        # Đảo ngược thuật toán VM:
        # Bước 1: Trừ đi hằng số K
        K = ((i * 0x045D9F3B) & M) ^ 0x9E3779B9
        h1 = ror((target - K) & M, b2 & 31)  # Bước 2: Dùng ROR thay cho ROL
        
        # Bước 3: Tìm ký tự answer[i]
        ans.append(((h_prev ^ h1) - b1) & 0xFF)
        h_prev = target
        
    return ans.decode('latin1')

if __name__ == '__main__':
    print(solve())
```


## 6. Bài học rút ra
- **Kỹ thuật mới học được:** Nhận diện tấn công **Trusting Trust (Compiler Backdoor)** - *"Đừng bao giờ tin tưởng tuyệt đối vào một compiler không rõ nguồn gốc, vì nó có thể tự chèn backdoor vào chính mã nguồn an toàn của bạn"*. Ngoài ra bài này đòi hỏi kỹ năng trích xuất bộ nhớ động (dump memory at runtime) qua GDB để đi tắt qua khâu decryption và kỹ năng Reverse một Custom VM thông qua việc đảo ngược thuật toán (Math Inversion).
- **Cách phòng chống:** Có thể dùng Diverse Double-Compiling (DDC) để kiểm tra tính toàn vẹn của trình biên dịch, tránh sử dụng các binary build toolchain không minh bạch hoặc bị nghi ngờ sửa đổi.
